## Part 2: Document Authoring Platform Selection (30 minutes)

### Platform Options

**Option 1: Microsoft SharePoint** ☐
- **Pros:**
  - ABC likely has Microsoft 365 licenses
  - IT team familiar with SharePoint admin
  - OneDrive sync for offline work
  - Integration with existing Microsoft tools
  - Strong permissions model
  - Audit trails
- **Cons:**
  - Can be slower than alternatives
  - Some complexity in setup
  - Permissions can be intricate
- **Cost:** Included with M365 (if ABC has it)
- **Best For:** Orgs already on Microsoft 365

**Option 2: Google Drive** ☐
- **Pros:**
  - Excellent real-time collaboration
  - Fast and responsive
  - Simple, intuitive interface
  - Good commenting/suggestion features
  - Easy sharing
  - Mobile apps excellent
- **Cons:**
  - Requires Google Workspace licenses
  - Another platform to manage
  - IT security review may be needed
  - Less control over permissions
- **Cost:** ~$12-18 per user per month (Google Workspace)
- **Best For:** Teams valuing collaboration and speed

**Option 3: Adobe Author Bus (Xwalk)** ☐
- **Pros:**
  - Native AEM integration
  - Purpose-built for Edge Delivery
  - Best preview experience
  - Direct publish from authoring
  - Included with AEM license
- **Cons:**
  - New tool for team
  - Learning curve
  - Less familiar than Word/Docs
  - Relatively new (fewer features)
- **Cost:** Included with AEM Cloud Service
- **Best For:** Teams wanting tight AEM integration

### Decision Criteria

| Criteria | SharePoint | Google Drive | Adobe Author Bus | Weight |
|----------|-----------|--------------|------------------|--------|
| Familiar to authors | High | High | Low | High |
| Collaboration features | Medium | High | Medium | High |
| Existing licenses | ? | ? | Yes (AEM) | High |
| IT security approval | Easy | Medium | Easy | High |
| Performance | Medium | High | High | Medium |
| AEM integration | Medium | Low | High | Medium |
| Cost | Low/None | Medium | None | High |

## Part 7: Agile Methodology & Sprint Planning (30 minutes)

### Sprint Structure

**Sprint Duration:**
☐ 1-week sprints  
☐ 2-week sprints (recommended)

**Rationale for 2-week:**
- Enough time to complete meaningful work
- Not too long to adapt to changes
- Industry standard

**Sprint Ceremonies:**

**1. Sprint Planning** (4 hours for 2-week sprint)
- When: First day of sprint
- Who: Full team
- Output: Sprint backlog, sprint goal

**2. Daily Standup** (15 minutes)
- When: Every day, same time
- Who: Development team
- Format: What did I do? What will I do? Any blockers?

**3. Sprint Review/Demo** (2 hours)
- When: Last day of sprint
- Who: Team + stakeholders
- Output: Demonstrated working software

**4. Sprint Retrospective** (1.5 hours)
- When: After sprint review
- Who: Team only
- Output: Improvement actions

**5. Backlog Refinement** (Ongoing)
- When: Mid-sprint
- Who: Product owner + team
- Output: Refined stories for next sprint

**ABC Preferences:**
- Sprint duration preference: ___________
- Ceremony timing preferences: ___________
- Current agile experience: ___________

### User Story Format

**Story Template:**
```
As a [role]
I want [capability]
So that [benefit]

Acceptance Criteria:
- Given [context]
  When [action]
  Then [expected result]
```

**Example:**
```
As a content author
I want to add a hero block to a page
So that I can create engaging landing pages

Acceptance Criteria:
- Given I'm editing a page in Google Docs
  When I add a hero block table with image, headline, text, CTA
  Then the hero block displays correctly on preview and production
- Hero block is responsive on mobile, tablet, desktop
- Lighthouse score remains 90+
```

**Story Pointing:**
- Fibonacci scale: 1, 2, 3, 5, 8, 13
- Relative sizing
- Team velocity established after 2-3 sprints

### Definition of Done

**A story is "Done" when:**
- [ ] Code written and follows standards
- [ ] Code reviewed and approved
- [ ] Tests passed (linting, Lighthouse)
- [ ] Functionality works as expected
- [ ] Responsive (mobile, tablet, desktop)
- [ ] Accessible (WCAG AA)
- [ ] Browser tested (Chrome, Firefox, Safari, Edge)
- [ ] Deployed to staging and validated
- [ ] Documentation updated (if needed)
- [ ] Accepted by product owner
- [ ] Merged to main and deployed to production
