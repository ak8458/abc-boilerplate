## Persona Example 1: Content Author (Marketing)

**Persona Name:** Sarah - Marketing Content Author  
**Role/Title:** Senior Content Specialist  
**Department:** Marketing & Communications  
**Number of Users:** 8-10  
**System(s) Used:** ☑ Edge Delivery Services (document authoring)

### Demographic Information
- **Experience Level:** 5 years in content marketing
- **Technical proficiency:** ☑ Intermediate
- **Current CMS:** WordPress (3 years)
- **Document tools:** Expert in Microsoft Word and Google Docs

### Goals & Motivations
**Primary Goals:**
1. Update website content quickly without developer help
2. Maintain brand consistency across all content
3. Collaborate with teammates on content review

**Success Metrics:**
- Publish content updates within same business day
- Zero brand compliance errors
- Positive feedback from website visitors

**Motivations:**
- Wants to be more agile and responsive
- Takes pride in creating excellent content
- Frustrated by bottlenecks

### Current State (Pain Points)

**Daily Tasks:**
1. Update product information pages
2. Create blog posts
3. Update campaign landing pages
4. Find and use approved images
5. Coordinate content approvals

**Current Workflow:**
1. Draft content in Word
2. Email to developer with request
3. Wait 2-5 days for developer availability
4. Review developer's implementation
5. Request changes if needed
6. Wait again
7. Finally published

**Pain Points:**
1. **Pain Point:** Dependent on developers for simple text changes
   - **Impact:** High - Can't respond quickly to market needs
   - **Frequency:** Daily
   - **Workaround:** None, just wait

2. **Pain Point:** Can't preview content before it goes live
   - **Impact:** High - Errors sometimes make it to production
   - **Frequency:** Weekly
   - **Workaround:** Try to catch in review, but it's hard

3. **Pain Point:** Finding the right images is time-consuming
   - **Impact:** Medium - Wastes 1-2 hours per project
   - **Frequency:** Daily
   - **Workaround:** Digging through shared folders, asking colleagues

### Future State (With AEM)

**How AEM Helps:**
1. Document-based authoring - familiar Word/Google Docs interface
2. Preview before publish - see exactly how it will look
3. Content Hub - find approved assets in seconds
4. Self-service - no developer dependency

**Desired Workflow:**
1. Open page document (in familiar tool)
2. Edit content using blocks
3. Preview in browser
4. Submit for approval (if needed)
5. Publish directly to website
6. Confirm it's live

**Expected Benefits:**
- Time to publish: From 2-5 days to same-day
- Quality: Fewer errors due to preview
- Satisfaction: More control and autonomy

**Must-Have Features:**
1. Easy-to-use content blocks (like tables in Word)
2. Real-time preview
3. Image picker integrated with Content Hub
4. Simple publish button

### System Interaction

**Frequency of Use:** Daily, 3-4 hours/day

**Primary Use Cases:**

**Use Case 1:** Update Product Page
- **Frequency:** Weekly
- **Steps:**
  1. Open product page document
  2. Update pricing or features using Cards block
  3. Preview page
  4. Publish
- **Success Criteria:** Updated in under 30 minutes

**Use Case 2:** Create Blog Post
- **Frequency:** 2-3 times per week
- **Steps:**
  1. Create new document
  2. Add hero image, body text, images
  3. Add CTA block at end
  4. Preview, publish
- **Success Criteria:** Create and publish in 2 hours

### Training & Support Needs

**Current Skills:**
- Microsoft Word: ☑ Expert
- HTML: ☐ Basic (can read simple HTML)
- Image editing: ☐ Basic (crop, resize)

**Learning Style:**
- Preferred: ☑ Hands-on workshop with practice
- Time available: 4 hours initial training
- Pace: ☑ Fast learner

**Training Needs:**
1. Block-based authoring (tables format)
2. Preview and publish process
3. Working with images
4. When to use which blocks

**Support Needs:**
- Preferred: ☑ Slack channel for quick questions
- Expectation: Response within 1-2 hours
- Style: ☑ Self-service preferred, but available help

### Quotes
> "I spend more time waiting for developers than actually creating content. I just want to update a page myself!"

> "Finding the right image feels like looking for a needle in a haystack. I wish I could just search and find it."

## Persona Example 2: DAM Administrator

**Persona Name:** Michael - DAM Administrator  
**Role/Title:** Digital Asset Manager  
**Department:** Marketing Operations  
**Number of Users:** 2-3 administrators  
**System(s) Used:** ☑ AEM Assets ☑ Content Hub (admin)

### Demographic Information
- **Experience Level:** 8 years in digital asset management
- **Technical proficiency:** ☑ Advanced
- **DAM experience:** Previous: Bynder, Widen
- **Technical skills:** Good understanding of metadata, taxonomy, APIs

### Goals & Motivations
**Primary Goals:**
1. Organize 200K+ assets logically
2. Ensure brand compliance
3. Make assets discoverable for all users
4. Maintain system performance and security

**Success Metrics:**
- Average asset find time < 2 minutes
- 100% brand-approved assets in system
- 80%+ user adoption rate
- Zero security incidents

### Current State (Pain Points)

**Current Workflow (File Server):**
1. Assets uploaded to shared drive with inconsistent naming
2. Manually organize into folders
3. No metadata, no search
4. Version control via filenames (image_v1, image_v2, image_final, image_final_v2...)
5. Share via email or links to specific folders

**Pain Points:**
1. **Pain Point:** No search capability - only browse folders
   - **Impact:** High - Users waste hours finding assets
   - **Frequency:** Daily
   - **Workaround:** Help users over email/Slack

2. **Pain Point:** Duplicate assets everywhere
   - **Impact:** High - Wasting storage, using wrong versions
   - **Frequency:** Constant
   - **Workaround:** Manual cleanup, but new duplicates appear

3. **Pain Point:** Can't enforce metadata or naming standards
   - **Impact:** Medium - Inconsistent, hard to manage
   - **Frequency:** Daily
   - **Workaround:** Guidelines doc that nobody reads

### Future State (With AEM Assets)

**How AEM Helps:**
1. Powerful search with metadata filters
2. Metadata schema with required fields
3. Smart tags (AI-powered)
4. Approval workflows
5. Usage tracking and analytics

**Desired Workflow:**
1. User uploads asset
2. Metadata auto-applied (smart tags, EXIF)
3. User completes required metadata
4. Submits for approval
5. DAM admin approves/rejects
6. Asset available in Content Hub
7. Track usage via reports

**Must-Have Features:**
1. Folder structure and permissions
2. Metadata schema configuration
3. Approval workflow management
4. Content Hub admin console
5. Usage reports and analytics

### System Interaction

**Frequency of Use:** Daily, 6-8 hours/day

**Primary Use Cases:**

**Use Case 1:** Configure Folder Structure and Permissions
- **Steps:**
  1. Design folder taxonomy
  2. Create folders in AEM Assets
  3. Set permissions per folder
  4. Test with users
- **Success Criteria:** Logical, secure, easy to navigate

**Use Case 2:** Review and Approve Assets
- **Frequency:** Daily
- **Steps:**
  1. Check approval queue
  2. Review asset and metadata
  3. Check brand compliance
  4. Approve or reject with comments
- **Success Criteria:** Process 20+ assets/day, 2-day SLA

**Use Case 3:** Run Usage Reports
- **Frequency:** Monthly
- **Steps:**
  1. Generate report on top downloaded assets
  2. Identify unused assets
  3. Archive or delete unused assets
  4. Share insights with team
- **Success Criteria:** Actionable insights for content strategy

### Training & Support Needs

**Current Skills:**
- DAM systems: ☑ Expert
- Metadata/taxonomy: ☑ Expert
- Basic scripting: ☑ Intermediate

**Training Needs:**
1. AEM Assets admin console deep-dive (16 hours)
2. Workflow configuration
3. Content Hub administration
4. API access and integration

**Support Needs:**
- Direct line to Adobe support for platform issues
- Admin documentation and best practices
- Community/forum access

### Quotes
> "A DAM is only as good as its metadata. I need to be able to enforce standards and make assets findable."

> "I need visibility into what assets are being used, what's not, and by whom. Data-driven decisions."
